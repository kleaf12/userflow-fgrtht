# Userflow

`Userflow` — внутренняя корпоративная платформа для управления операциями, сервисными процессами и решениями руководства


![2026-03-12 16 22 15](https://github.com/user-attachments/assets/b09410b7-4cdc-4c30-aebe-76668a165fd4)

Проект решает рабочие задачи:

- управление активами, заявками, оргструктурой и базой знаний;
- единый интерфейс для операционных процессов и мониторинга статусов;
- быстрая работа с данными через структурированный API-слой;
- предсказуемое качество релизов через quality gate (`typecheck/lint/format/build`) и CI.

Кто использует проект:

- руководители направлений и тимлиды: смотрят дашборды, бюджет, риски и SLA по процессам;
- service desk / operations / support: ведут заявки, активы, инциденты и операционные статусы;
- HR и руководители команд: используют оргструктуру и карточки сотрудников;
- security/compliance: контролируют события и задачи по безопасности;
- DevOps/инженерные команды: отслеживают автоматизации и интеграции систем;
- разработка и QA: развивают продукт, проверяют качество изменений и релизную стабильность.

## Стек

- `React 18` (библиотека для построения интерфейсов из компонентов).
- `TypeScript` (надстройка над JavaScript со статической типизацией).
- `Vite` (быстрый dev-сервер и сборщик проекта).
- `Ant Design` (UI-библиотека готовых компонентов: таблицы, формы, кнопки, модалки).
- `React Query` (управление серверным состоянием: загрузка, кеширование, обновление данных).
- `React Router` (роутинг: переходы между страницами в SPA).
- `Sass` (расширение CSS: переменные, вложенность, миксины).

## Что нужно перед стартом

Установите:

- `Node.js` (рекомендуется LTS 20+)
- `npm`
- `Git`

В редакторе (например, VS Code) установите расширения: (_и перезапустите vscode_)

- `ESLint` (от Microsoft)
- `Prettier - Code formatter` (от Prettier)

Тогда ошибки линтера и форматирования будут видны прямо во время разработки.

Проверьте версии:

```bash
node -v
npm -v
git --version
```

## Быстрый запуск


```bash

npm run format        # форматирование prettier

npm run check         # полный прогон quality gate (typecheck+lint+format+build)
```

## CI в Pull Request

При открытии или обновлении PR GitHub Actions автоматически запускает пайплайн `.github/workflows/ci.yml`.

В PR отображаются отдельные статусы проверок:

- `Typecheck` (`npm run typecheck`)
- `Lint` (`npm run lint`)
- `Format Check` (`npm run format:check`)
- `Build` (`npm run build`)

Если любой шаг падает, PR считается неготовым к ревью до исправления.

## Архитектура (FSD)

```text
src/
  app/       # инициализация приложения, роутинг, провайдеры
  pages/     # страницы (например: dashboard, profile, settings)
  widgets/   # крупные блоки страницы (например: Header, Sidebar, UserTable)
  features/  # пользовательские действия/сценарии (например: LoginForm, UpdateAvatar, CreateTask)
  entities/  # бизнес-сущности (например: User, Task, Project с model/api/ui)
  shared/    # общий код: ui-kit, api-клиент, config, utils, хуки
```

## Имитация backend

В проекте сейчас используется моковый backend (имитация API), чтобы можно было разрабатывать UI без реального сервера.

- Основной файл: `src/shared/api/mockApi.ts`
- Все страницы получают данные через этот слой (как будто это настоящий backend).

## Ежедневный рабочий сценарий разработчика

1. Подтянуть изменения:

```bash
git pull
npm install
```

2. Создать ветку под задачу:

```bash
git checkout -b feature/<short-task-name>
```

3. Сделать задачу, проверить качество: (чтобы не было красных ошибок - error)

```bash
npm run typecheck
npm run lint
npm run build
npm run format:check
```

Что проверяют команды:

- `typecheck` — что типы TypeScript корректны.
- `lint` — что код соответствует правилам ESLint.
- `build` — что проект собирается в production-режиме.
- `format:check` — что форматирование соответствует Prettier.

Как фиксить проблемы:

- ошибки `lint`: `npm run lint:fix` (часть исправится автоматически, остальное вручную);
- ошибки `format:check`: `npm run format`;
- ошибки `typecheck/build`: исправлять код по тексту ошибки (тип, импорт, несуществующее поле и т.д.).

4. Закоммитить изменения:

```bash
git add .
git commit -m "feat: кратко что сделано" (на англ.)
```

5. Отправить ветку и открыть PR:

```bash
git push -u origin feature/<short-task-name>
```

Как открыть PR:

1. Перейдите в репозиторий на GitHub.
2. Нажмите `Compare & pull request` для отправленной ветки.
3. Проверьте `base` (обычно `main`) и `compare` (`feature/<short-task-name>`).
4. Заполните заголовок и описание: что сделано, как проверялось, что посмотреть ревьюеру.
5. Нажмите `Create pull request`.

## Правила сдачи задач

- Пишем понятные имена переменных/функций на английском.
- Не оставляем закомментированный и неиспользуемый код.
- В строках используем только двойные кавычки (`"`), одинарные считаются ошибкой линтера.
- Импорты должны быть отсортированы по категориям, между категориями обязателен пустой ряд (за нарушение `eslint` выдаёт ошибку).
- `console.log` и другие `console.*` запрещены (за нарушение `eslint` выдаёт ошибку).
- Перед PR проект должен проходить `typecheck`, `lint`, `build`.
- Один PR = одна логическая задача.

## Частые проблемы

`npm install` падает:

- проверьте версию `Node.js`;
- удалите `node_modules` и `package-lock.json`;
- снова выполните `npm install`.

Порт занят:

Почему это происходит:

- другой проект уже запущен на `5173`;
- предыдущий dev-сервер не завершился корректно;
- иногда порт может занять системный процесс.

Как избежать:

- перед новым запуском остановите старый dev-сервер (`Ctrl + C` в терминале);
- не держите несколько Vite-проектов на одном порту;
- при необходимости запускайте сразу на другом порту.

```bash
npm run dev -- --port 5174
```

Ошибки после `git pull`:

- проверьте, что вы на нужной ветке (`git branch`);
- при конфликтах сначала решите их, потом повторите проверки (`typecheck/lint/build`).
